## 关键词对齐修复：whisper 同音字误转写导致对不上

- 问题：narration 写 `读作fú`/`句首发语词`，whisper 转写成 `读作福`/`巨手发语词`（同音字），导致在 whisper 文本里搜关键词字符串不准。
- 修复 `align_keywords`：不再在 whisper 文本里直接搜关键词。改为：
  1. 在**无损的 narration 原文**里确定关键词字符位置（优先 `[词]` 标记 positions，否则取中部出现）；
  2. 新增 `_align_text()` 用 `difflib.SequenceMatcher` 把 narration 原文字符级对齐到 whisper 转写文本，相等块逐字符映射、差异块线性插值兜底；
  3. whisper 字符位置 → 字符级时间轴（按 word.start 展开到每个字）→ 时间戳。
- 配套：`_save_json` 不再 pop `_positions`，保留到 script.json/timeline.json，重跑对齐可精确去歧义。
- 注意：现有 script.json 缺 `_positions`（旧版已 strip），要拿到精确定位需重新生成 narration（带 `[词]` 标记）；仅重跑对齐也已比旧版稳健（不受同音字影响）。

## 升级：用 edge-tts 词级边界(WordBoundary)替代 whisper 做对齐主路径

- 洞察：`[词]` 标记只解决"关键词在文本哪个位置"，不解决"位置→时间"；时间一直靠 whisper 重新听写 TTS 音频，whisper 同音字误转写是准确度瓶颈。
- 正解：edge-tts 合成时 `Communicate.stream()` 会推送 `WordBoundary` 事件（含 `offset`/`duration`，单位 100ns=1e7 分之一秒），是引擎自报的每个词发音真值，零 ASR 误差。
- 改动（build_timeline.py）：
  1. `synthesize_audio` 改用 `comm.stream()`，边写 mp3 边收集 boundaries 返回；TTS 阶段把边界存到 `audio/{idx:02d}.words.json`。
  2. 新增 `align_by_boundaries()`：拼接边界词文本→字符时间轴，difflib 对齐 narration↔边界文本（同源仅标点差异，近乎完美），定位关键词。
  3. 新增 `_narration_pos()` 抽出关键词原文定位逻辑，`align_keywords`/`align_by_boundaries` 共用。
  4. 对齐阶段优先用 words.json 走 boundary，缺失才 fallback whisper；TTS 跳过条件改为 mp3+words.json 都在，缺边界自动重合成补全。
- 重跑：现有诫子书 mp3 缺 words.json 会自动重合成补边界，直接 `python script/build_timeline.py "诫子书 全文"` 即可（需在项目根，先删 timeline.json 以触发重对齐）。

## 更正（同日）：WordBoundary 方案作废——edge-tts 中文不返回词边界

- 实测：`comm.stream()` 对中文文本（zh-CN-YunxiNeural）返回的 WordBoundary 数量恒为 0，是 edge-tts 逆向微软 Read-Aloud 接口的已知限制。上面"用 WordBoundary 替代 whisper"的方案不成立，已全部回退（`synthesize_audio` 恢复 `comm.save`，删除 `align_by_boundaries` 及 words.json 相关逻辑，对齐统一走 `align_keywords`）。保留了重构出的 `_narration_pos()` 公共函数。
- 关键结论：对齐准确度低的**真正主因不是同音字**（difflib 已兜底），而是**现有 `data/timelines/诫子书/script.json` 是旧代码产物，无 `[词]` 标记、无 `_positions`**（grep `_positions` = 0 处）。导致"夫/之/行"等讲解中多次出现的字只能"取中部出现"瞎猜而飘位。
- whisper 逐字时间戳质量其实不错（每字独立 start/end），时间精度不是瓶颈，文本定位才是。
- 解决：必须重新生成带 `[词]` 标记的 narration（删 script.json 重跑），positions 链路才生效，对齐才准。当前代码生成阶段已带标记指令、parse_markers 解析、_save_json 保留 _positions，链路完整，只是旧数据未走过。

## 决定：对齐主路径改为线性估时 align_by_ratio（用户选 B 方案）

- 用户目的明确：只要"讲解音频讲到某关键词时，原文句中该词高亮"（player.html 已实现：句中关键词包 `<span class="word" data-word>`，按 keyword_timings[].start 设 setTimeout 调 highlightKeyword 加 .active 黄底）。准度只取决于 keyword_timings 的 time。
- 新增 `align_by_ratio(narration, keywords, duration, positions)`：`time ≈ 关键词字符位置/总字数 × 音频时长`。edge-tts 中文语速均匀，误差 1~2s，对"高亮词"足够，零 ASR、零依赖、秒级。定位仍走 `_narration_pos`（优先 [词] 标记 positions）。
- build_timeline.py：对齐循环默认走 align_by_ratio（无 duration 则 read_mp3_duration 补）；新增 `--use-whisper` 切回 align_keywords 作兜底。打印"对齐时间轴（线性估时）"。
- 仍需重生成带标记数据：`rm script.json timeline.json && python script/build_timeline.py "诫子书 全文"` 再 build_player。

## build_pptx.py 终于修好「导出视频带旁白」

- 根因：之前只写 `<p:transition><p:snd>`（切换音效，导出视频不输出）和悬空的 `<p:sndTgt>` timing；幻灯片 spTree 里根本没有真正的嵌入媒体对象。PowerPoint 导出视频只采集「幻灯片上自动播放的嵌入媒体」，故必然静音。
- 修复 `_embed_audio_post`：在每页 spTree 注入真正的 `<p:pic>` 音频媒体对象（pic 用 `off=-3000000` 移到画面外，不显示图标），配标准 `mediacall` 自动播放 timing（spid=1000+slide_num），`transition advTm` 控制每页时长=旁白时长。
- 每页 rels 需三条关系指向同一 wav/png：`audio`(给 `<a:audioFile r:link>`) + `media`(给 `<p14:media r:embed>`，namespace p14=.../powerpoint/2010/main，ext uri `{DAA4B4D4-6D71-4841-9C94-3DE7FCFB9230}`) + `image`(占位 1x1 透明 png)。Content_Types 注册 wav。
- MP3→WAV 用 pcm_s16le 44100 立体声。python-pptx 可正常重开不触发修复。用户在 PowerPoint：文件→导出→创建视频 即得带声 MP4。

## PowerPoint 导出视频声音诊断

- 用户反馈导出 MP4 没声音；检查 `data/timelines/诫子书/slides.mov`，ffprobe 显示有 2 个流：H.264 视频 + AAC 音频，音频 duration≈1048s，volumedetect `mean_volume=-24.7 dB`、`max_volume=-2.8 dB`，不是静音轨。
- 结论：当前已导出的 `slides.mov` 文件本身带声音；若用户听不到，更可能是播放端/文件版本/转换 MP4 时丢音轨，而不是 PPT 未导出音轨。
